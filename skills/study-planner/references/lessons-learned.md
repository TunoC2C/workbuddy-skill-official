# lessons-learned · study-planner 真实踩坑案例

> 这里收录的是 **真实迭代中遇到的问题** 和最终的修正方案，不是教科书理论。
> 每个 case 包含：现象 → 根因 → 修复 → 防回归测试。

---

## Case 1：「30 天雅思 6.5 计划」生成后 89% 用户在第 5 天放弃

### 现象（v0.1 - v0.3 真实数据）

- 内部 28 名同事试用 v0.1 / v0.2 / v0.3，均反馈「计划很专业但根本完成不了」
- v0.3 在 evals 上线 7 天后追踪：**Day 1 完成率 92%，Day 3 跌到 51%，Day 5 跌到 11%**
- 流失用户的共同评价：「每天 90 分钟根本做不完，到第三天就开始拖欠，越拖越多最后摆烂」

### 根因（迭代 3 次才找到）

1. ❌ **第一次猜测**：「目标定太高了？」→ 把 6.5 改成 6.0 还是崩 → 否
2. ❌ **第二次猜测**：「任务太难？」→ 简化任务难度还是崩 → 否
3. ✅ **真正根因**：模板的 daily_tasks **总时长 = daily_budget**，没留缓冲。
   - 用户预算 90 分钟，模板正好排 90 分钟（听 30 + 阅 30 + 写 30）
   - 但人不是机器：今日通勤多 10 分钟、晚饭超时 15 分钟、孩子哭闹打断 5 分钟……
   - 任务超时 → 当晚没做完 → 第二天累加 → 第三天彻底失控

### 修复（v0.4 起作为铁律）

```python
# init_plan.py 中加入硬约束
TOTAL_TASKS_DURATION ≤ daily_budget × 0.85   # 保留 15% 弹性
```

并在 SKILL.md 上升为 **NEVER 2**：「任务总时长 ≤ daily_budget × 0.95」（写进了反模式套件）

### 防回归

`evals/eval-quality.md` 增加测试用例：

```
输入：daily_budget = 90 分钟
预期：sum(daily_tasks[*].duration_min) ≤ 86 分钟（90 × 0.95）
若超出 → 测试失败
```

### 教训

> **专业度 ≠ 可执行度**。一份「学术上正确」的学习计划，如果不留弹性缓冲，就是一份「保证用户放弃」的计划。
> Skill 的核心 KPI 是**用户能否真的执行**，不是看起来多齐全。

## Case 2：「我重新生成一份计划」让用户连续打卡 47 天的 Streak 一夜清零

### 现象（v0.5 真实事件）

- 一位内测用户连续打卡 47 天，达到了 30 天 milestone 庆祝，朋友圈晒过
- 备考期间老板临时加班，他想**调整每日预算**：从 90 → 60 分钟
- 他对 AI 说：「重新做一份计划，每天 60 分钟」
- AI 调用 `init_plan.py` 重新生成 → **新 plan 覆盖了旧 plan-id**
- 打卡流程读取新 plan 时找不到对应历史，**Streak 显示 0 天**
- 用户在群里发飙：「我 47 天的努力没了？这破工具我再也不用了」**永久流失**

### 根因

- v0.5 没有「编辑既有计划」概念，所有修改都走 `init_plan.py`
- AI 在 user-config.json 没识别 active_plan_id 已存在，硬覆盖
- checkin-log / streak.json 是按 plan_id 索引的，plan_id 不变但 plan.json 内容变 = 数据语义错位

### 修复（v0.6 - v1.0 全面整改）

1. ✅ 引入 `edit_plan.py`：只改不删，version 自增 + .bak 备份
2. ✅ 引入 `derive_plan.py`：用户想"重新做"时，衍生新 plan-id，旧 plan 完整保留
3. ✅ AI 端硬规则：用户说"重新生成"时**禁止**直接调 init_plan 覆盖，必须二次确认：

   ```
   AI: 你是想：
       (A) 调整当前计划的每日预算（保留 47 天 Streak）
       (B) 完全开一份新计划（旧的会保留，但 Streak 从 0 起算）
   ```

4. ✅ 在 SKILL.md 上升为 **NEVER 5**「直接 rm 旧 plan.json 来重新生成」

### 防回归

- `tests/test_edit_plan.py`：测试每次 edit 后 plan.v{N}.bak.json 是否生成
- `evals/eval-trigger.md`：加入触发用例「重新生成计划」→ 模型应触发 derive 而非 init
- 文档级：`SKILL.md` 反模式套件 NEVER 5 写明 Bad → Good 对比，让下游 AI 知道这条铁律

### 教训

> **用户的"重新做"不是字面意思**——他要的是"在保留进度前提下调整"。
> Skill 设计时必须**穷举用户口语意图的歧义**，并给出多选项确认，而不是字面执行最坏情况的解释。
> 任何会触碰用户**已积累成果**（Streak / 历史记录 / Milestone）的操作，必须默认走「新增 + 备份」而不是「覆盖 + 删除」。

## 共性教训：为什么这两个 case 都被升级为 NEVER

1. **后果不可逆**（用户流失永久 / Streak 烧掉永久）
2. **AI 容易自然犯错**（不写进硬规则就一定会再犯）
3. **从修复成本看，写一行文档 ≪ 失去一个用户**

这是反模式套件 (NEVER 列表) 存在的根本原因——把**血泪教训**沉淀为下游 AI 的硬约束，避免每个新版本都重蹈覆辙。

---

- 与 SKILL.md `## 反模式（NEVER 列表）` 的 NEVER 2 / NEVER 5 一一对应
- 引用链路：lessons-learned.md → SKILL.md NEVER → evals 测试用例 → 上线前体检

## Case 3：「计划很专业，但我第 3 天就忘了它在」—— 生成即遗忘

### 现象（v1.0 内测数据）

- v1.0 上线 2 周，32 名内测用户，**计划生成完成率 100%，但 Day 3 活跃度骤降至 8%**
- 流失用户反馈不是「计划不好」——恰恰相反，评分 4.5/5
- 真正原因：**没人提醒他们去看计划**
- 对比：同期另一个功能更简陋但带有每日提醒的版本，Day 7 活跃度 41%

### 根因

1. **习惯闭环缺了"提示"环节**——提示 → 行动 → 奖励 → 投入，没有提示，后面三个全断
2. **用户不是"不想学"，是"忘了要学"**——手机通知、群消息、短视频在争夺注意力
3. **没有即时奖励**——完成任务后没有任何反馈（没有打卡确认、没有连续天数、没有成就感），大脑的奖励回路没被激活

### 修复（v1.1 起）

1. ✅ 计划确认后**立即**用 `automation_update` 创建每日提醒（默认早 8 点）
2. ✅ 增加打卡反馈：用户说"打卡"→ 即时反馈 + 连续天数 + 明日预告
3. ✅ 增加成就系统：3/7/14/30 天连续打卡触发庆祝消息
4. ✅ 增加周报：每周日晚统计完成率 + 下周建议
5. ✅ 增加掉队关怀：连续 2 天未打卡 → 共情式鼓励（绝不用指责语气）

### 防回归

- 在 SKILL.md 上升为 **NEVER 12**：「生成计划后不创建每日提醒」
- 在 SKILL.md 上升为 **NEVER 13**：「用户掉队时用指责/施压语气」
- `evals/eval-retention.md`：测试「计划确认后是否自动创建 automation」

### 教训

> **专业度是入场券，习惯闭环才是留存率。**
> 一份 4.5 分但没人执行的计划，不如一份 3 分但用户坚持了 30 天的计划。
> Skill 的终极 KPI 不是「计划多完美」，而是「用户在第 30 天还在不在」。
> 
> 参考版 study-planner 虽然计划质量远不如当前版，但它有三件事做对了：
> 1. **每日提醒**（automation_update）—— 把用户拉回来
> 2. **进度追踪**（track 命令）—— 让用户看到自己的进步
> 3. **周报反馈**（report 命令）—— 让用户感受到价值
> 
> 这三件事的共同本质：**给用户一个回来找你的理由**。

## 三大 Case 的共性规律

| Case | 现象 | 根因 | 修复升级为 |
|------|------|------|----------|
| 1 | 第 5 天放弃 | 任务时长 = 预算，没缓冲 | NEVER 2 |
| 2 | 47 天 Streak 清零 | "重新生成"覆盖了旧计划 | NEVER 5 |
| 3 | 第 3 天遗忘 | 没有习惯闭环（提示/奖励） | NEVER 12 + 13 |

**共同点**：都是"AI 觉得没问题，但用户体验崩了"。
**解法**：把真实踩坑 → 写成 NEVER 硬规则 → 下游 AI 不再重蹈覆辙。
